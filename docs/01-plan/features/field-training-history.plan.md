# [Plan] 현장실습 다회차 기록 관리 및 최종 취업처 분리

## 1. 개요
현재 시스템은 학생당 1회의 현장실습 정보만 기록할 수 있으며, '최종 취업처'와 '현장실습 업체'가 동일한 필드(`company`)를 공유하고 있습니다. 이를 개선하여 현장실습 이력을 다회차로 관리하고, 실습 결과(채용전환/복교)를 체계적으로 추적하며, 최종 취업 정보를 독립적으로 관리할 수 있도록 확장합니다.

## 2. 주요 목표
- **다회차 실습 이력 관리**: 1:N 관계의 실습 이력 테이블(`field_training_records`) 도입.
- **최종 취업처 분리**: `student_employments` 테이블의 `company`는 '최종 확정 취업지'로 정의하고, 실습 업체는 이력 테이블에서 관리.
- **결과 추적 강화**: 각 실습 회차별로 '채용전환' 또는 '복교' 여부와 사유를 필수로 기록.
- **통합 관리 UI**: `/students` 페이지에서 요약 정보 확인 및 상세 이력 관리 모달 제공.

## 3. 세부 추진 계획

### 3.1 데이터베이스 구조 개편 (DB Migration)
- **신규 테이블 생성**: `field_training_records`
  - `id` (PK), `student_id` (FK), `order` (차수), `company` (실습업체), `period` (시작/종료일), `outcome` (결과: 진행중/채용전환/복교), `conversion_date`, `return_reason` 등.
- **기존 테이블 수정**: `student_employments`
  - 실습 관련 중복 필드(`start_date`, `end_date`, `is_hiring_conversion`, `is_returned`) 제거.
  - `company` 필드는 '최종 취업처' 용도로 유지.

### 3.2 데이터 이관 (Data Migration)
- 기존 `student_employments`에 저장된 실습 데이터를 `field_training_records`의 '1차' 데이터로 안전하게 복사.

### 3.3 로직 및 API 개발
- **데이터 조회 (`src/lib/data.ts`)**: 학생 정보 조회 시 최신 실습 이력을 Join하여 가져오는 로직 추가.
- **서버 액션 (`src/app/students/actions.ts`)**: 
  - 실습 이력 추가/수정/삭제 액션 구현.
  - 실습 결과가 '채용전환'일 경우 최종 취업처(`student_employments.company`) 자동 업데이트 제안 로직.

### 3.4 UI/UX 구현 (`/students`)
- **메인 테이블 컬럼 조정**: 
  - 현재 실습 상태(가장 최근 차수)를 요약해서 보여주는 컬럼 구성.
  - 실습 상세 관리를 위한 [이력] 버튼 추가.
- **실습 이력 관리 모달**:
  - 타임라인 형태의 실습 리스트 UI.
  - 새 실습 회차 추가 및 기존 회차 수정 기능.

## 4. 기대 효과
- **정확한 통계**: 실습 업체와 최종 취업처가 다른 경우에도 정확한 데이터 산출 가능.
- **이력 추적**: 학생이 취업에 성공하기까지 거쳐간 모든 실습 과정과 복교 사유 분석 가능.
- **데이터 무결성**: 과정(실습)과 결과(취업)를 분리하여 데이터 구조의 논리적 명확성 확보.
