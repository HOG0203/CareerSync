## 1. 개요 (Overview)

본 개편은 CareerSync 시스템이 학교 현장의 실제 학사 일정(학사학년도)과 교직원 업무 분장(담당 학반)에 완벽하게 부합하도록 근본적인 구조를 재설계한 작업입니다. 데이터 무결성 확보를 위한 DB 물리적 분리부터, 직관적인 연도 필터링, 그리고 유연하면서도 강력한 데이터 보안 정책(RLS) 적용이 포함되었습니다.

## 2. 주요 개선 사항 (Key Improvements)

### 2.1. 데이터베이스 구조 정규화 (DB Normalization)
*   **배경**: 기존 단일 테이블(`student_employments`) 구조는 인적사항과 취업 현황이 혼재되어 데이터 중복 및 RLS 권한 분리에 어려움이 있었습니다.
*   **변경 사항**:
    *   `students` (학생 기본 인적사항 마스터 테이블)와 `student_employments` (취업 및 현장실습 현황 테이블)로 1:1 분리.
    *   분리 과정에서 누락되었던 필수 컬럼(`career_aspiration`, `special_notes`) 추가 보완 완료.
    *   두 테이블은 `id`(UUID)를 공유하여 완벽한 데이터 정합성 유지.

### 2.2. 학사학년도 중심 연도 계산 로직 도입 (Academic Year Logic)
*   **배경**: 기존 시스템은 '졸업연도'를 기준으로 데이터를 조회하여, 현재 재학 중인 학생들을 직관적으로 관리하기 어려웠습니다.
*   **핵심 공식**: `학년(Grade) = 4 - (졸업연도(GradYear) - 학사학년도(BaseYear))`
    *   *(예: 시스템 기준 학사학년도가 2026년일 때, 2027년 졸업 예정자는 3학년으로 계산됨)*
*   **적용 범위**:
    *   **대시보드 및 통계**: 필터 및 제목이 "0000학년도 3학년" 형식으로 표기되도록 직관화. (초기값 자동 매칭)
    *   **학적 이력(Academic History) 자동 동기화**: 신규 등록, 일괄 업로드, 진급 처리, 정보 수정 시 위 공식을 통해 해당 데이터의 졸업연도에 맞는 '정확한 과거/현재 학사학년도'를 역산하여 이력(`student_academic_history`)을 기록함.

### 2.3. 교직원 맞춤형 권한 세분화 및 보안 강화 (RLS & Access Control)
*   **배경**: 선생님들이 전교 통계를 볼 수 없었고, RLS 무한 재귀 참조로 인해 시스템 전체 조회가 차단되는 치명적 오류가 존재했습니다.
*   **해결 및 고도화**:
    *   `is_admin()` 단일 검증 함수로 통합 및 `SECURITY DEFINER` 적용하여 무한 재귀(Infinite Recursion) 원천 차단.
    *   **권한 모델 재설계**:
        *   **관리자 (Admin)**: 제한 없는 전체 접근, 수정, 삭제 권한.
        *   **담임 교사 (Teacher)**:
            *   **조회(SELECT)**: 대시보드 및 취업 통계 활용을 위해 전교생 데이터 열람 허용.
            *   **수정(UPDATE/INSERT)**: 오직 본인 프로필(`profiles`)에 설정된 담당 학년도, 학과, 반에 속한 학생만 수정 가능하도록 데이터베이스 레벨(RLS) 및 UI 레벨 동시 차단.
